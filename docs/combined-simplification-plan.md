## Core Insight

Both simplification docs describe the same architectural problem from opposite ends:

```
Frameworks (producer) ──> Rho.Comms (transport) ──> LiveView (consumer)
```

The frameworks doc asks how to stop the producer from coupling to the transport.
The web doc asks how to stop the consumer from parsing transport artifacts.
They're one migration with a shared dependency.

The BAML spike adds a third dimension: structured output parsing can be unified
under a single Rust-based engine (`baml_elixir`) that works with any model,
replacing both `ReqLLM.generate_object` (frameworks) and `StructuredOutput` +
`SchemaCoerce` (TypedStructured).

## Schema Strategy: Zoi as Source of Truth

### Why Zoi

Zoi is a schema-as-data library — no macros,
just composable functions. A single Zoi schema definition produces:

1. An Elixir struct with enforced keys
2. Dialyzer typespecs via `Zoi.type_spec/1`
3. Validation via `Zoi.parse/2` for LLM response verification
4. A BAML class definition via `RhoBaml.SchemaCompiler.to_baml_class/1`

This avoids maintaining parallel definitions in `.baml` and Elixir.

### What an LLM Function Module Looks Like

```elixir
defmodule RhoFrameworks.LLM.RankRoles do
  use RhoBaml.Function,
    client: "OpenRouter",
    params: [query: :string, role_list: :string, limit: :int]

  @schema Zoi.struct(__MODULE__, %{
    indices: Zoi.array(Zoi.integer()) |> Zoi.describe("1-based indices of most similar roles"),
    reasoning: Zoi.string() |> Zoi.describe("Brief explanation")
  })

  @enforce_keys Zoi.Struct.enforce_keys(@schema)
  defstruct Zoi.Struct.struct_fields(@schema)
  @type t :: unquote(Zoi.type_spec(@schema))

  @prompt ~S"""
  You are a role matching assistant. Given a query and a numbered list of
  existing role profiles, return the indices of the most similar roles.

  Return at most {{limit}} indices.
  Only return numbers that appear in the list.

  Query: {{query}}
  Roles: {{role_list}}

  {{ ctx.output_format }}
  """

  # Generated by `use RhoBaml.Function`:
  # def call(args, opts \\ [])    — sync call, returns {:ok, %__MODULE__{}} | {:error, term()}
  # def stream(args, cb, opts)    — streaming call
end
```

### `use RhoBaml.Function`

Not a macro-heavy system. A thin `__before_compile__` hook that:

1. Reads `@schema` and `@prompt` module attributes
2. Converts Zoi schema → BAML class string via `RhoBaml.SchemaCompiler`
3. Writes the `.baml` file (class + function) to the app's `priv/baml_src/`
4. Defines `call/2` and `stream/3` that:
   - Delegate to `BamlElixir.Client.call/3` or `.sync_stream/4`
   - Validate result with `Zoi.parse(@schema, result)`
   - Return the struct

### `RhoBaml.SchemaCompiler`

Converts Zoi schemas to BAML class definitions (~30 lines):

| Zoi | BAML |
|-----|------|
| `Zoi.string()` | `string` |
| `Zoi.integer()` | `int` |
| `Zoi.float()` | `float` |
| `Zoi.boolean()` | `bool` |
| `Zoi.array(Zoi.integer())` | `int[]` |
| `Zoi.optional()` | `type?` |
| `Zoi.describe("...")` | `@description("...")` |
| `Zoi.map(%{...})` | nested `class { ... }` |

### Where Zoi Applies vs Where It Doesn't

| Use case | Zoi as source of truth? | Why |
|----------|------------------------|-----|
| Frameworks static schemas | **Yes** | Schema defined once, generates struct + `.baml` |
| Future LLM functions | **Yes** | Same pattern scales — add a module, get struct + BAML |
| TypedStructured dynamic | **No** | Tool_defs come from NimbleOptions at runtime |
| BAML client configs | **No** | Infrastructure, hand-written `.baml` files |

## App Ownership: Each App Owns Its Schemas

```
apps/rho_baml/                           # Pure library — no priv/, no configs
├── lib/
│   ├── rho_baml.ex                      # helpers (baml_path/1, etc.)
│   ├── rho_baml/function.ex             # `use RhoBaml.Function` hook
│   ├── rho_baml/schema_compiler.ex      # Zoi → BAML class conversion
│   └── rho_baml/schema_writer.ex        # tool_defs → dynamic .baml (added in Phase 4)
└── mix.exs                              # deps: baml_elixir, zoi

apps/rho_frameworks/                     # Owns its schemas AND client configs
├── lib/rho_frameworks/llm/
│   ├── rank_roles.ex                    # Zoi schema + @prompt + use RhoBaml.Function
│   ├── score_lens.ex                    # same pattern
│   └── semantic_duplicates.ex           # same pattern
└── priv/baml_src/                       # Everything BAML needs in one tree
    ├── clients/
    │   ├── openrouter.baml              # cheap DeepSeek
    │   └── anthropic.baml               # Claude for quality tasks
    └── functions/                       # Generated at compile time by RhoBaml.Function
        ├── rank_roles.baml
        ├── score_lens.baml
        └── find_semantic_duplicates.baml

apps/rho/                                # Owns its dynamic schemas
└── priv/baml_src/                       # TypedStructured's BAML tree
    ├── clients/                         # Own copy of client configs
    │   ├── openrouter.baml
    │   └── anthropic.baml
    └── dynamic/                         # Generated at runtime by SchemaWriter
        └── action_<hash>.baml
```

`rho_baml` is a pure library — no `priv/`, no configs. It provides:
- `RhoBaml.Function` behaviour (compile-time Zoi → BAML for static schemas)
- `RhoBaml.SchemaCompiler` (Zoi → BAML class string conversion)
- `RhoBaml.SchemaWriter` (runtime tool_defs → BAML for TypedStructured, added in Phase 4)

Each consumer app owns its full BAML tree: client configs, schemas, and
generated functions. Adding a new LLM function to frameworks means adding
a module to `rho_frameworks/lib/`, not touching `rho_baml`. Client configs
are duplicated per consumer app (they're ~5 lines each).

## What Becomes Dead Code

**Original plan estimate (pre-audit):** ~911 LOC across `StructuredOutput`,
`SchemaCoerce`, and `ActionSchema.render_prompt/1`.

**Post-audit reality (2026-04-26):** ~40 LOC. Most of the "replaced by BAML"
claim is wrong — BAML only handles parsing inside the agent loop, not on the
LV display side or for Direct-strategy tool args.

### Actual dead code (safe to delete)

| Symbol | Lines | Production callers |
|--------|-------|--------------------|
| `Rho.ActionSchema.parse_and_dispatch/3` | ~15 | None — `TypedStructured` uses `dispatch_parsed/3` (typed_structured.ex:71). Only tests reference it. |
| `Rho.ActionSchema.render_prompt/1` | ~25 | None — BAML emits `{{ctx.output_format}}` instead. Only tests reference it. |

Removing `parse_and_dispatch/3` also drops the only `apps/rho/` call site of
`Rho.StructuredOutput.parse`.

### Live code that the plan mislabeled as dead

| Module | Lines | Why it's still live |
|--------|-------|--------------------|
| `Rho.StructuredOutput` | 537 | LV display side parses streaming/replay text from tape (4 sites: `stream_envelope.ex`, `chat_components.ex`, `chat_overlay_component.ex`, `projections/session_state.ex`). BAML produces parsed maps inside the agent loop, but the LV consumers re-read raw text from tape and need their own partial-JSON parser. |
| `Rho.SchemaCoerce` | 374 | Primary coercer for **all** tool args via `Rho.ToolArgs.cast/2` → `SchemaCoerce.coerce_fields/3` (tool_args.ex:102). Also called by `ActionSchema.dispatch/3` (action_schema.ex:243), which both `parse_and_dispatch` and `dispatch_parsed` route through. The phase4 notebook already corrected this inline. |

`ActionSchema.build/1` and `ActionSchema.dispatch/3` remain unchanged. BAML
still feeds `dispatch_parsed/3` with a parsed map; `dispatch/3` continues to
do discriminant matching, field coercion (via `SchemaCoerce`), and intent
classification.

### Legacy Worker path — removed (2026-04-26)

`Rho.Agent.Worker.build_turn_opts_legacy/3` and `run_turn_legacy/6` are gone.
Rather than migrate every `Supervisor.start_worker` caller, `Worker.init`
now synthesizes a `%Rho.RunSpec{}` via `build_default_run_spec/2` whenever
`:run_spec` isn't supplied — folding any legacy spawn-time opts
(`:tools`, `:system_prompt`, `:model`, `:max_steps`) into the spec. With
`state.run_spec` always non-nil, `start_turn/3` no longer branches on it
and the legacy path is deleted. Existing callers (`AppLive`,
`SessionLive`, `MultiAgent`) keep working unchanged.

RunSpec without code changes. `Rho.Runner.run/3` (legacy 3-arity) stays
for direct test usage of Runner.

`StructuredOutput` (537 LOC) and `SchemaCoerce` (374 LOC) **stay**. Both are
live code — see "What Becomes Dead Code" section above for the audit.

## Risk Assessment

| Risk | Mitigation |
|------|-----------|
| BAML file I/O on every TypedStructured turn | Profile; `.baml` files are tiny (~500 bytes), local SSD write |
| baml_elixir NIF binary compatibility | Precompiled for darwin-aarch64 + linux; CI verified |
| BAML version churn (pre-release) | Pin exact version; NIF API is stable (call/stream) |
| Collector logs contain API keys | Don't persist Collector logs; use only for debugging |
| Zoi → BAML type mapping gaps | Start with primitives (string, int, float, bool, array, optional); extend as needed |
| Dual LLM calling systems (ReqLLM + BAML) | Clean separation: ReqLLM for agent loop (Direct strategy), BAML for structured output |
| Zoi API changes | Zoi is at 0.18.0 (pre-1.0); pin version, converter is ~30 lines to update |
