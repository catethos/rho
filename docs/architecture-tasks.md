# Architecture Refactor — Task List

Implementation tasks for [architecture-review.md](architecture-review.md), split into phases sized for a single LLM context window.

Each phase produces a working system. No phase depends on a later one being incomplete.

---

## Phase 1: Contract & Plugin Split

**Goal:** Stabilize the runtime contract and break up the god module. No new abstractions.

**Reference:** architecture-review.md §Problem 6 (Context Threading), §Problem 1 (arg casting), §Problem 2 (God Plugin)

- [x] **1.1** Change `tool_def` type from `execute: fn args -> result` to `execute: fn args, context -> result`
  - Update `Rho.Plugin` type spec
  - Update `TurnStrategy.Direct` to pass `runtime.context` as second arg
  - Update all existing `execute` fns in `RhoFrameworks.Plugin` to accept and ignore `_ctx` for now
  - Verify: `mix test` passes, agent tools still execute

- [x] **1.2** Add schema-driven arg casting in `TurnStrategy.Direct`
  - Added `Rho.ToolArgs.cast/2` — casts only declared param keys to atoms
  - Integrated into Direct, Structured, Worker, LiteWorker
  - Simplified `args["key"] || args[:key]` → `args[:key]` across all stdlib tools
  - Verify: tools receive atom-keyed args correctly

- [x] **1.3** Extract tool modules from `RhoFrameworks.Plugin`
  - Created `RhoFrameworks.Tools.LibraryTools` — 14 library tools
  - Created `RhoFrameworks.Tools.RoleTools` — 7 role tools
  - Created `RhoFrameworks.Tools.LensTools` — 3 lens tools
  - Created `RhoFrameworks.Tools.SharedTools` — 1 shared tool
  - Each module exports a `tools(context)` function returning `[tool_def]`
  - Reduced `Plugin.ex` from 1266 lines to ~90 lines (thin aggregator)
  - Verify: `mix test` passes, 25 tools unchanged, agent behavior identical

## Phase 2: SignalRouter Refactor

**Goal:** Refactor SignalRouter into a clean reducer + effects pipeline, creating the seam needed for later phases.

**Reference:** architecture-review.md §Problem 5 (State Split)

- [x] **2.1** Extract pure reducer functions from `SignalRouter.route/3`
  - `update_session_state/2` — pure SessionState.reduce, collects effects
  - `update_workspace_states/3` — pure per-workspace projection.reduce, tracks changed keys
  - `update_shell/3` — pure Shell.record_activity + maybe_auto_open, collects pulse effects
  - Each stage takes socket + signal, returns socket with updated assigns

- [x] **2.2** Extract impure effects into an explicit `apply_effects/1` step
  - Session effects (push_event, send_after) and workspace effects (pulse decay timers)
    collected as `_effects` assign during pure stages
  - `apply_effects/0` applies all collected effects via `SessionEffects`, then cleans up
    temporary pipeline assigns (`_effects`, `_changed_ws_keys`)

- [x] **2.3** Rewrite `SignalRouter.route/3` as a pipeline
  ```
  signal = enrich_signal(signal, socket)
  socket |> init_pipeline() |> update_session_state(signal) |> update_workspace_states(signal, registry) |> update_shell(signal, registry) |> apply_effects()
  ```
  - Verify: `mix compile --warnings-as-errors` passes, signal routing behavior identical

## Phase 5: Effect Dispatch

**Goal:** Wire effect structs (created in Phase 3) to the web layer so tools can drive UI without direct coupling.

**Reference:** architecture-review.md §Problem 3 (Tool-to-UI Coupling), §Migration Strategy

  - Pattern-match clauses for `%Rho.Effect.Table{}`, `%Rho.Effect.OpenWorkspace{}`
