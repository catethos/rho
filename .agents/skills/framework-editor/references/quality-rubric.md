# Quality Rubric for Behavioral Indicators

Rules for writing high-quality proficiency level descriptions.

## Must Be Observable

Bad: "Has knowledge of project management methodologies"
Good: "Applies Agile methodology to plan sprints, prioritize backlog items, and facilitate daily standups"

Bad: "Understands customer needs"
Good: "Conducts structured interviews with 3+ stakeholders to identify and document unmet customer needs"

## Must Use Action Verbs

Start every indicator with a Bloom's taxonomy verb appropriate to the proficiency level. See `dreyfus-model.md` for level-appropriate verbs.

Bad: "Is aware of security best practices"
Good: "Follows OWASP security guidelines when reviewing code for common vulnerabilities"

## Must Be Non-Overlapping

Adjacent levels must be clearly distinct. If level 3 says "troubleshoots issues independently" then level 2 should NOT say "resolves problems with minimal help" — too similar.

**Differentiation levers:**
- **Scope:** task → project → team → department → organization → industry
- **Autonomy:** with guidance → independently → leads others → sets standards
- **Complexity:** routine → non-routine → ambiguous → novel → unprecedented
- **Impact:** individual → team → organization → industry

## Must Include Context

Bad: "Communicates effectively"
Good: "Communicates technical decisions to non-technical stakeholders using clear analogies and visual aids"

The context specifies: who, what, when, how.

## Length

- 1-2 sentences per indicator
- Max 40 words per sentence
- One clear observable behavior per indicator (not a list of 5 things)

## Enterprise Language

- Match the formality level to the domain (corporate HR vs startup vs government)
- Avoid jargon that only one department would understand
- Use consistent terminology across all skills in the framework
